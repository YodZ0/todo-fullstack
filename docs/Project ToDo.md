# Техническое задание — Проект «ToDo»

**Версия:** 1.1 (MVP)
**Тип:** Pet-проект / персональный инструмент
**Дата:** 21.07.2026

---

## 1. Обзор проекта

### 1.1. Проблема

ToDo-приложений много, но в каждом чего-то не хватает, и функции разбросаны по разным сервисам. Цель — собрать нужные функции «всё в одном месте»: ведение задач, учёт рабочего времени по задачам через таймер и просмотр статистики/активности.

### 1.2. Цель и определение успеха

Готовый проект, развёрнутый на собственном VPS, которым автор пользуется в повседневной жизни. Формальных бизнес-метрик нет. Критерий готовности — работающий MVP на VPS, доступный автору и его друзьям бесплатно.

### 1.3. Референс и отличие

- **Главный референс:** TickTick.
- **Ключевое отличие:** бесплатное решение «всё в одном» для себя: задачи + таймер учёта времени + статистика активности.

### 1.4. Тип продукта

- **MVP:** веб-приложение, однопользовательское (личная продуктивность).
- **Перспектива:** мобильное приложение.

---

## 2. Пользователи и сценарии

### 2.1. Целевой пользователь

Человек, ведущий личную продуктивность.

### 2.2. Основные сценарии

- **Сценарий 1 (задача + таймер):** создать задачу → запустить таймер по задаче → выполнить работу → остановить таймер → отметить задачу выполненной.
- **Сценарий 2 (быстрый список):** создать несколько задач → отмечать их выполненными по мере закрытия.

## 3. Функциональные требования

### 3.1. Аутентификация и аккаунты

- Регистрация и вход по **email + пароль**.
- Вход через **Google OAuth 2.0**.
- **Полное удаление аккаунта** пользователем со всеми связанными данными (задачи, разделы, теги, интервалы времени, настройки).
- Пароли хранятся только в виде хеша (bcrypt/argon2), в открытом виде не хранятся и не логируются.

### 3.2. Задачи (Tasks)

**Обязательные атрибуты:**

| Атрибут   | Тип    | Описание                 |
| --------- | ------ | ------------------------ |
| `title`   | string | Название задачи          |
| `is_done` | bool   | Выполнена / не выполнена |
| `date`    | date   | Дата задачи              |
| `user`    | ref    | Владелец задачи          |

**Необязательные атрибуты:**

| Атрибут            | Тип   | Описание                                                                               |
| ------------------ | ----- | -------------------------------------------------------------------------------------- |
| `description`      | text  | Описание                                                                               |
| `priority`         | enum  | Приоритет: `none` / `low` / `medium` / `high`                                          |
| `recurrence`       | rule  | Повтор: ежедневно, еженедельно, ежемесячно, ежегодно, каждый рабочий день, произвольно |
| `recurrence_until` | date  | Дата завершения повторов                                                               |
| `tags`             | ref[] | Теги — **несколько на задачу** (создаёт пользователь)                                  |
| `section`          | ref   | Раздел (напр. «Учёба», «Работа» — создаёт пользователь)                                |

**Операции:** создание, редактирование, удаление, отметка «выполнено/не выполнено».
**Статусы:** только бинарные — «сделано / не сделано».
**Подзадач нет.**

### 3.3. Разделы и теги

- **Разделы** — создаются пользователем, группируют задачи (одна задача — один раздел).
- **Теги** — создаются пользователем; **задаче можно назначить несколько тегов**.

### 3.4. Повторяющиеся задачи

- Типы повтора: **ежедневно, еженедельно, ежемесячно, ежегодно, каждый рабочий день, произвольно**.
- Поддержка **даты завершения повторов** (`recurrence_until`).
- Правило повтора хранится в формате iCal **RRULE**.
- **Стратегия генерации:** хранить правило повтора и **материализовать N ближайших** экземпляров задачи (по мере закрытия/приближения генерировать следующие).
- **Каскадное удаление серии:** при удалении **последней активной** задачи серии (последней с `is_done=false`) выполняется массовое удаление всей связанной серии повторяющихся задач.

### 3.5. Представления

- **Список** — простой перечень задач.
- **Календарь** — задачи по дате (`date`).
- Kanban-доска в объём **не входит**.

### 3.6. Таймер / учёт времени

- Режим: **простой старт/стоп**, привязан **к конкретной задаче** (свободных активностей без задачи нет).
- Одновременно активен **только один таймер** на пользователя.
- **Ручное добавление и редактирование** интервала времени.
- **Лимит неактивности** (напр. 4 часа, настраивается): при неактивности дольше лимита таймер авто-останавливается.
- **Поведение при закрытии (web):** при возвращении, если есть незавершённый таймер, приложение спрашивает, что делать (продолжить / остановить и сохранить / отбросить). Состояние таймера хранится на сервере. _Точный UX — см. §10._
- **Поведение (mobile, перспектива):** таймер продолжает идти в фоне.
- Биллинговых ставок нет.

### 3.7. Напоминания и уведомления

- **MVP:** напоминания по **email**.
- **Перспектива (mobile):** push.
- Канал и включение уведомлений **настраиваются пользователем**.
- Для email-напоминаний нужен планировщик/воркер (§6).

### 3.8. Статистика

- **Линейный график:** время / количество задач.
- **Сетка активности за месяц** (heatmap в стиле GitHub).
- **Периоды:** неделя, месяц, произвольный диапазон.
- Экспорт в MVP не требуется. **Перспектива:** PDF-отчёт.

## 4. Нефункциональные требования

- **Масштаб (MVP):** до 100 пользователей.
- **Производительность:** ответ API — **не более 2000 мс** (целевое значение существенно ниже).
- **Безопасность:** хеширование паролей; HTTPS; полное удаление аккаунта и данных; чувствительные данные не попадают в логи.
- **Бэкапы:** регулярные автоматические резервные копии PostgreSQL.
- **Аналитика поведения пользователей:** не требуется.
- **Accessibility:** специальных требований нет.

## 6. Архитектура и технологии

| Слой               | Технологии                                 |
| ------------------ | ------------------------------------------ |
| **Frontend**       | React, TypeScript, shadcn/ui, Tailwind CSS |
| **Backend**        | Python, FastAPI, Pydantic, SQLAlchemy      |
| **База данных**    | PostgreSQL                                 |
| **Инфраструктура** | VPS                                        |

**Следствия требований:**

- **Планировщик/воркер** для email-напоминаний и авто-остановки таймера по лимиту неактивности (APScheduler или Celery + Redis; на MVP можно начать с лёгкого варианта).
- **Отправка email** через внешний SMTP-провайдер.
- **Деплой:** VPS, рекомендуется Docker Compose (nginx + FastAPI + PostgreSQL), HTTPS через Let's Encrypt.
- **Бэкапы:** периодический `pg_dump` по расписанию с хранением копий.
- **Хранение данных:** на сервере (бэкенд с аккаунтами). Офлайн-синхронизация в MVP не нужна — появится с mobile.

## 10. Открытые вопросы

1. **Реализация таймера в целом** — отложено до слайса таймера (v0.0.9–v0.0.11).
2. **UX незавершённого таймера при возврате (web):**
   - (A) Модалка «Таймер идёт N ч M мин» → _Продолжить / Остановить и сохранить фактическое / Отбросить_ — _рекомендация_.
   - (B) Авто-стоп по моменту последней активности (last heartbeat) + уведомление с возможностью возобновить.
   - (C) Продолжать в фоне по умолчанию (как mobile), без диалога.
   - Связанный вопрос — точное определение «неактивности» для web (нет взаимодействия / нет heartbeat / закрытая вкладка).

_Решено:_ стратегия генерации повторов (правило + N ближайших, §3.4), уровни приоритета (§9), формат правила повтора RRULE.

## 11. Критерии приёмки MVP

- Регистрация (email/Google), вход и удаление аккаунта работают.
- Создание/редактирование/удаление задач со всеми атрибутами, разделами, несколькими тегами, включая повторяющиеся (с каскадным удалением серии).
- Задачи видны в «Списке» и «Календаре».
- Таймер по задаче запускается/останавливается; ручная корректировка интервалов; работает лимит неактивности.
- Приходят email-напоминания согласно настройкам.
- Доступна статистика: линейный график и месячная сетка активности с выбором периода.
- Тёмная тема и русский язык работают.
- Приложение развёрнуто на VPS по HTTPS, настроены регулярные бэкапы БД.
